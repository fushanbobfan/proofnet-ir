# v0.7 design: exact ProofNetEquivalent typed key and wire qualification

## Scope

The target relation is exactly `Certificate.ProofNetEquivalent`: a bounded
vertex reindexing followed by semantically irrelevant link-list permutation.
Ordered conclusions, tensor/par premise order, formula labels, and axiom
endpoint orientation remain significant. This design does not canonicalize
reordered conclusions and is not an arbitrary unlabeled-graph isomorphism
algorithm.

`CheckedCertificate.sameProofNet?` is already the exact production pairwise
decision boundary. v0.7 aims to add a stable single-value representation for
the same relation without weakening that theorem boundary.

## Phase 1: forward invariant

`proofNetCanonicalFamily` is an executable, finite, complete family
specification, but it enumerates link-list permutations. The first v0.7 phase
maps every family member to its existing v0.3 `equivalenceCanonicalString` and
selects the lexicographically least string:

```text
certificate
  -> proofNetCanonicalFamily
  -> map equivalenceCanonicalString
  -> lexicographic minimum
  -> Option String
```

The option type exposes the finite minimum operation directly. Lean proves the
candidate list is nonempty for every certificate, so the result is total.
Lean also proves that every successful result belongs to the serialized family
and that `ProofNetEquivalent` inputs have equal results.

This phase deliberately proves only:

```text
ProofNetEquivalent left right
  -> fingerprint? left = fingerprint? right
```

It does not assume that different normalized certificates cannot serialize to
the same string.

## Phase 2: equality-reflecting typed code

Lean's JSON library does not currently provide the exact
`Json.compress` injectivity or decoder-round-trip theorem needed to turn the
phase-1 string into a reverse-complete key without adding an assumption. v0.7
therefore uses a separate structural encoding:

- formulas, links, and certificate fields receive explicit constructor tags;
- every nested token sequence is length-framed;
- vertex indices and lengths use a transparent unary reference token;
- the certificate payload begins with
  `proofnet-structural-code-0.1`.

Lean proves the structural encoding injective for formulas, links, and complete
certificates. `proofNetCanonicalCode?` selects the lexicographically least
structural code in the same complete finite canonical family. The resulting
public theorem is:

```text
StructurallyWellFormed left
-> StructurallyWellFormed right
-> (ProofNetEquivalent left right
    <-> canonicalCode? left = canonicalCode? right)
```

`proofNetEquivalent_iff_canonicalCode_of_check` derives the premises from
checker acceptance. Thus the typed token code is an exact identity key for the
documented domain. The original JSON `proofNetCanonicalFingerprint?` remains a
forward invariant only; the project does not silently strengthen it through an
unproved serialization assumption.

## Wire and compatibility

The phase-1 value reuses v0.3 strings internally but does not change the v0.3
wire contract. The phase-2 typed code has an internal version marker but is not
yet a serialized wire contract. A public exact wire key must introduce a new
canonicalization marker and, if its payload changes, a new wire version. The
release gate includes native parsing, checked decoding, old-to-new migration
fixtures, malformed-input tests, property tests across reindexing/link
permutations, and a pinned clean downstream consumer.

No existing v0.2 or v0.3 string will be silently reinterpreted.

## Performance boundary

Both compact results currently require materializing the factorial
`proofNetCanonicalFamily`. Return size, logical completeness, and runtime
complexity are separate facts. The implementations are excluded from the
current CI performance budget and must not be described as production-grade.

Before release qualification, either:

- replace factorial materialization with a proved equivalent construction; or
- retain it with explicit size limits and measured budgets that match the
  intended research-library use.

The exact pairwise checker remains available regardless of the chosen
canonical construction.
