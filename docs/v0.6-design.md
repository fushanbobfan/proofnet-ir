# v0.6 persistent LeanProp bridge design

## Status and boundary

The bridge is a conservative new layer. It does not change `Formula`,
`Certificate`, Danos--Regnier switching correctness, serialization, proof-net
identity, or the two sequentialization theorems. Existing unit-free cut-free
MLL results retain exactly their v0.5 meaning.

The new judgment is indexed by two ordered proposition contexts:

```text
persistent ; linear |- Lean proposition
```

The contexts contain occurrences, not merely formula labels. A derivation is
a typed proof template, and `LeanProp.Derivation.toProof` interprets it as a
real Lean proof term. The Lean kernel is therefore the final semantic checker.

## Resource discipline

- `persistentAxiom` and `linearAxiom` begin with one resource occurrence.
- Binary introduction/elimination rules concatenate their input contexts.
- `persistentWeaken` explicitly discards a persistent occurrence.
- `persistentContract` explicitly supplies two uses from one persistent
  occurrence.
- exchange is explicit for both contexts.
- there is intentionally no linear weakening or linear contraction
  constructor.

`Derivation.linearAxiomCount_eq_length` proves that every derivation has
exactly one linear-axiom leaf per linear context occurrence. This turns the
exact-use discipline into kernel-checked metatheory rather than an inspection
of the constructor list.

The environment layer also proves append/split round trips and both exchange/
inverse-exchange round trips. `ContextPermutation` is complete for
proposition-level `List.Perm` under `Nonempty`, and both persistent and linear
exchange are admissible for every such permutation. The `Nonempty` boundary
is deliberate because Prop-valued permutation evidence cannot be eliminated
into Type-valued computational syntax.

Trust auditing distinguishes the claims: `Derivation.toProof` is axiom-free;
the exact-use and dependent-environment equality theorems use exactly
`propext`, because their type indices are proposition-valued lists.

The separate `LeanPropNormalization` module recursively cancels every
immediate `persistentContract (persistentWeaken premise)` redex. Its indexed
return type preserves the full judgment. Kernel theorems establish reduced
output, fixed points, idempotence, persistent-structural-size nonincrease,
linear-resource-count preservation, and pointwise `toProof` preservation.
This is a precise local structural normal form, not a quotient by all possible
commuting conversions, and it is a noncomputable proof-construction API rather
than a runtime transformation of raw schema values.

This design makes a shared persistent premise visible: two branches initially
produce two context occurrences, and a contraction node discharges the
duplication. It also prevents the bridge from silently treating a proof-net
occurrence as reusable.

## Supported proof-template nodes

The first bridge layer supports Lean conjunction introduction/elimination,
ordinary implication introduction/elimination, equality rewriting, universal
instantiation, and existential witness introduction. These are Lean
propositions, not a claim that `And` is MLL tensor or that another ordinary
connective is par.

The committed template corpus exercises persistent duplication and weakening,
linear context splitting, linear modus ponens, equality transport, universal
instantiation, and existential witnesses. Each example exports a theorem whose
body is reconstructed by the total interpreter.

A separate first-order schema calculus covers named atoms, ordinary
conjunction, and implication without embedding arbitrary Lean expressions in
data. Its deterministic corpus contains 600 positive templates in six strata.
`Schema.PackedDerivation.sound` reconstructs a Lean proof for every atom
valuation and uses exactly `propext` when eliminating its packed proposition
indices; the underlying `Derivation.toProof` interpreter remains axiom-free.
An unindexed `Schema.Raw.Derivation` boundary now checks rule applications and
infers their exact persistent context, linear context, and goal. Its errors
carry a stable category and child path. The theorem
`Raw.Derivation.infer?_ofIndexed` proves every erased indexed schema is
accepted with the original boundary and uses exactly `propext`; the lower-level
indexed-permutation erasure theorem is axiom-free. The separate raw-boundary /
typed-permutation agreement theorem uses exactly `propext`. CI exercises 600 erased positives
and 1,000 malformed inputs, covering all nine error categories plus nested
path propagation with exact expected diagnostics.
The separate `leanprop-schema-0.1` contract serializes every formula,
permutation, and raw derivation constructor in this fragment. Its native Lean
parser rejects unknown fields and versions, bounds recursive depth, and exposes
syntax only through `checkedFromString` when typed `elaborate?` succeeds. The
kernel theorem `inferAt_eq_elaborateAt` proves that formula-only inference and
typed elaboration have exactly the same result and diagnostics after forgetting
the indexed witness; `elaborate?_complete` lifts every `infer?` acceptance to
an indexed derivation with the same boundary. A checked wire value exposes
`toPacked` and `sound`, so external JSON reaches universal Lean proof
reconstruction rather than stopping at a Boolean gate. All 600 positives
round-trip and elaborate through canonical JSON, all 1,000 negatives preserve
their checker diagnostics through the wire path, two independent JSON Schema
fixtures are validated, and a deterministic 5,000-case mutation gate exercises
the native parser. Lean exports the 1,600 labeled wire records itself; CI pins
the stream to the SHA-256 manifest in `datasets/leanprop-v0.1/manifest.json`, so
Python does not reimplement acceptance labels. Equality and quantifier schemas are deliberately not
serialized until a type-safe external term language exists.

## Source basis

The two-context split follows the unrestricted/linear distinction recorded in
the complete Pfenning page audit, especially physical pages 32--34. That source
also warns that the contexts have distinct substitution principles. The
project-specific research plan stages this bridge after the verified MLL core.
Neither source licenses changing the existing proof-net certificate semantics.

## v0.6 release qualification

- `leanprop-schema-0.1` is the only LeanProp wire version in v0.6, so no
  cross-version migration exists yet. A second version is required to add
  explicit migration fixtures;
- no MLL/LeanProp connective identification is added in v0.6. Ordinary Lean
  conjunction/implication remain intentionally separate from MLL tensor/par;
  any future selected semantic interpretation requires its own design and
  theorems;
- the public `v0.6.0` tag is an annotated tag over the green release-candidate
  commit. A separate clean consumer resolves that tag from GitHub and
  typechecks `CheckedDerivation.inferred`, `toPacked`, `sound`, and the
  persistent normalizer API without a path dependency.
